# Bionic - Skills 運用ルール
> コマンドごとの担当と手順を定義する

---

## コマンド一覧

| コマンド | 主担当 | 副担当 | 目的 |
|---------|--------|--------|------|
| `/research` | Claude | Codex補助 | 市場・技術・競合の調査統合 |
| `/spec` | Claude | — | 何を作るか・成功条件・非目標 |
| `/plan` | Claude + Codex | — | 境界と優先順位（Claude）+ 技術具体化（Codex） |
| `/refine` | Codex | — | 仕様を実装可能な粒度に落とす |
| `/build` | Claude Code | — | 実装・git・MCP |
| `/test` | Claude Code | Codex（観点出し） | 実行・検証 |
| `/review` | Codex | — | バグ・セキュリティ・仕様ズレ |
| `/ship` | Claude | — | 出すかどうかの最終判断 |

---

## 各コマンドの詳細

### `/research`
**主担当：Claude**

目的：外の世界の変化を把握し、判断の材料を作る。

手順：
1. トピックを設定する
2. 一次情報を優先して調査・要約する
3. 重要度スコアをつける
4. `research/` に保存する
5. 重要なものは `docs/` にサマリーを残す

成果物：
- `research/` 配下のメモ
- 必要に応じて `docs/` のサマリー

---

### `/spec`
**主担当：Claude**

目的：何を作るかを決める。作らないことも決める。

手順：
1. ビジョンとの整合を確認する（`docs/BIONIC_PRODUCT.md`）
2. 成功条件を定義する（何ができたら完了か）
3. 非目標を明示する（何をやらないか）
4. `docs/` に仕様書として残す

成果物：
- `docs/` 配下の仕様書
- `CURRENT.md` の更新

---

### `/plan`
**主担当：Claude（境界・優先順位） + Codex（技術具体化）**

目的：どう作るかを決める。Claudeが何をやるかを決め、Codexがどう作るかを具体化する。

手順：
1. Claude が実装の境界・優先順位・done条件を決める
2. Codex がファイル分割・state・API・DB・型を具体化する
3. Claude が最終確認して承認する
4. `CURRENT.md` に反映する

成果物：
- `CURRENT.md` の更新
- 必要に応じて `docs/` の技術設計メモ

---

### `/refine`
**主担当：Codex**

目的：仕様の曖昧な部分を潰して実装可能な粒度に落とす。`/plan` の前段として使う。

手順：
1. 仕様書を読む
2. 曖昧語・非目標の不明確な箇所を洗い出す
3. Claude に確認が必要な部分を「要Claude確認」として列挙する
4. 実装可能な粒度に言語化する

成果物：
- 具体化された仕様メモ
- 「要Claude確認」リスト

---

### `/build`
**主担当：Claude Code**

目的：実装する。

手順：
1. `CURRENT.md` と `AGENTS.md` を読む
2. `WORK_LOG.md` に開始エントリを書く
3. 実装する
4. `pnpm build && pnpm typecheck` を通す
5. セルフレビューを行う
6. `WORK_LOG.md` に完了エントリを書く
7. `git commit && git push` まで行う
8. コミットハッシュを報告する

成果物：
- 実装済みコード
- `WORK_LOG.md` の完了エントリ
- コミットハッシュ

---

### `/test`
**主担当：Claude Code / 観点出し：Codex**

目的：動くことを証明する。

手順：
1. 必要に応じて Codex が「何を壊しやすいか」の観点を先に出す
2. Claude Code がその観点を元にテストを実行する
3. 失敗した場合は `WORK_LOG.md` に記録して `/build` に戻る
4. 通過したら完了を報告する

成果物：
- テスト結果レポート
- `WORK_LOG.md` の記録

---

### `/review`
**主担当：Codex**

目的：壊れていないか・漏れていないかを確認する。

確認観点：
1. 動かないコード・バグ
2. セキュリティ漏洩リスク
3. 公開前に危険な点
4. README通りの再現性
5. 期待仕様とのズレ

ルール：
- findingがなければ「問題なし」と明言する
- 設計変更の提案は最小限にし、必要な場合は「要Claude確認」と明記する

成果物：
- findingリスト（優先度付き）または「問題なし」

---

### `/ship`
**主担当：Claude**

目的：出すかどうかを判断する。

手順：
1. Phase 1の成功条件を確認する
2. Codexのレビューが完了しているか確認する
3. 出す・凍結する・後回しにするを判断する
4. `WORK_LOG.md` に判断理由を残す

成果物：
- リリース判断の記録
- `CURRENT.md` の更新

---

## 原則

**Spec before code**
コードを書く前に仕様を決める。`/spec` なしに `/build` に入らない。

**Small, atomic tasks**
一度に一つのことをやる。大きすぎるタスクは `/plan` で分割する。

**One slice at a time**
一つのスライスを完全に終わらせてから次に進む。

**Tests are proof**
テストは証拠。「動いていると思う」ではなく「テストが通っている」を基準にする。

**Smaller is safer**
完璧を待つより、小さく動くものを早く届ける。
