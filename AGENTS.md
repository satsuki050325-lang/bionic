# Bionic - エージェント行動規範
> 全エージェントが作業開始前に必ず読むこと

---

## 第一指示：いいものを作る

これは全指示の中で最優先される。

「いいもの」の定義:
- 一線で活躍するエンジニアが作るような品質
- 世の中の需要・課題を見越した先進的な設計
- 企業が出すプロダクトと同等の品質・信頼性

この基準を常に意識すること:
- 実装コストを理由にした妥協は却下
- 技術的負債・セキュリティ・保守性・拡張性を常に検討する
- 「今は動く」ではなく「正しく動く」を目指す
- Codexのレビューで指摘された問題は必ず修正する
- 妥協案を提示する時は「これは妥協である」と明示する

## 役割分担

| エージェント | 役割 |
|------------|------|
| Claude（チャット） | 戦略・仕様・判断・リサーチ統合 |
| Codex | 技術設計の具体化・レビュー・品質ゲート |
| Claude Code | 実装・ファイル操作・git操作・MCP |

---

## 作業開始前に必ず読むこと

1. `CURRENT.md`（今の状態・次の1手・指示書）
2. `AGENTS.md`（このファイル）
3. `docs/BIONIC_PRODUCT.md`（プロダクト思想）
4. `docs/TECHNICAL_DESIGN.md`（技術設計）
5. UIを実装・変更する場合は `docs/DESIGN.md` を必ず参照すること

---

## Claudeの役割

- 戦略・仕様・優先順位の最終判断
- `/research` `/spec` `/ship` の主導
- `/plan` の最終責任者（境界と優先順位を決める）
- チャットが変わる時は `CURRENT.md` に文脈を残す

**Claudeに投げるもの**
- 設計判断・優先順位・「何を作るべきか」
- プロダクト思想との整合確認
- リサーチ結果の統合と判断

---

## Codexの役割

- `/plan` の技術具体化を担当（最終責任者はClaude）
  - Claudeが決めた境界・優先順位をファイル・状態・API・DB・型へ落とす
- `/refine` `/review` `/test観点` の主導
- 「どう壊れるか」の洗い出し

**Codexに投げるもの**
- ファイル分割・state・API・DB・型の具体化
- バグ・セキュリティ・再現性のレビュー
- 仕様を実装可能な粒度に落とす作業

**Codexの報告優先順位**
1. 動かない
2. 漏れる・セキュリティリスク
3. 公開すると危ない
4. README通りに再現できない
5. 期待したUXと実装がズレている

findingがなければ「問題なし」と明言すること。

---

## Claude Codeの役割

- `/build` `/test` の主導
- 実装・ファイル操作・git操作・MCP操作

**Claude Codeの標準手順**
1. `CURRENT.md` と `AGENTS.md` を読む
2. `WORK_LOG.md` に開始エントリを書く
3. 対象コードを確認する
4. 実装する
5. `pnpm build && pnpm typecheck` を通す
6. セルフレビューを行い見落としを報告する
7. `WORK_LOG.md` に完了エントリを書く
8. `git commit && git push` まで行う
9. コミットハッシュを報告する

---

## 推奨開発フロー

```
1. Claude で方針を決める（/spec /research）
2. Codex で技術設計を具体化する（/plan /refine）
3. Claude Code で実装する（/build）
4. Claude Code でテストする（/test）
5. Codex でレビューする（/review）
6. Claude でリリース判断する（/ship）
```

## 依頼フォーマット

### Codexへのレビュー依頼
```
変更は加えず、以下の観点だけ見てください。
- バグ・動かないコード
- セキュリティ漏洩
- 公開前に危険な点
- README通りの再現性
- 期待仕様とのズレ

findingがなければ「問題なし」と明言すること。
```

### Claude Codeへの実装依頼
```
実装して、build確認して、セルフレビューして。
WORK_LOG.mdに完了エントリを記録すること。
git commit && git push まで行うこと。
```

## 不在時ルール

**Claude不在時**
- Codex・Claude Codeは設計変更しない
- 必要なら「要Claude確認」として暫定対応にする
- `docs/BIONIC_PRODUCT.md` `docs/TECHNICAL_DESIGN.md` は触らない

**Codex不在時**
- レビューを省略してよいが「Codexレビュー待ち」と明記する
- ship判断までは進めない

**Claude Code不在時**
- 実装案まで作成可。ファイル変更は最小限または保留

## タスク開始前の確認

新しいタスクを始める前に必ずIDEAS.mdを確認すること。
そのタスクに関連する「参照タイミング」が記載されているアイデアがあれば、
CURRENT.mdの実装計画に反映してから着手する。

確認の手順:
1. CURRENT.mdで次のタスクを確認する
2. IDEAS.mdを開いて「参照タイミング」を検索する
3. 該当するアイデアがあればClaude（設計担当）に共有する
4. Claudeが実装計画に反映したことを確認してから実装を開始する

## 作業記録ルール

1. 作業開始前に `WORK_LOG.md` に開始エントリを書く
2. 作業終了時に完了エントリを書く（判断・未解決・既知リスクを含む）
3. Claude以外は設計変更時に必ず「要Claude確認」を付ける
